### Architectural Standards Established This Session

**Log Level Policy — apply to all future audits:**

| Event Type | Required Level |
|-----------|----------------|
| Normal operational info (loaded, opened, closed) | `logger.info` |
| Position rejection (R:R, risk limits) | `logger.warning` |
| DB or external service errors | `logger.warning` |
| Circuit breaker trigger | `logger.warning` |
| Unrecoverable startup errors | `logger.error` |

**Parallel File Consistency Rule:** `armed_signal_store.py` and `watch_signal_store.py` are sibling files that must mirror each other exactly in structure, log levels, error handling patterns, and import style. Any change to one must be cross-checked against the other.

**DB Connection Pattern** (must be present in every DB-accessing function):
```python
conn = None
try:
    conn = get_conn()
    cursor = dict_cursor(conn)
    # ... work ...
    conn.commit()
finally:
    if conn:
        return_conn(conn)
```

**Import Order Standard** (reference: `position_manager.py`):
1. `from utils import config`
2. stdlib (`datetime`, `zoneinfo`, `logging`, `time`, `typing`)
3. `logger = logging.getLogger(__name__)` immediately after `import logging`
4. `from app.*` imports
5. Optional `try/except` dependency blocks

### Critical Find: Issue #55 — Postgres UPDATE Bug in explosive_mover_tracker.py

`update_override_outcome()` uses `UPDATE ... ORDER BY timestamp DESC LIMIT 1`. PostgreSQL does **not** support `ORDER BY` or `LIMIT` in `UPDATE` statements — this is SQLite-only syntax. On Postgres this will either raise a syntax error or silently update all matching `PENDING` rows for a ticker rather than the most recent one. This is the only **MED-severity** new issue from Phase 3. Fix in Issue #55.

Every analytics file uses `logger.info` on error/exception paths. The log level policy (established Session 2) requires `logger.warning` for all DB and external service errors. This affects 9 issues (#50–60) but is a bulk cosmetic fix — a single commit can sweep all analytics files.
