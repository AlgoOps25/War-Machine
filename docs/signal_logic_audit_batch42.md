Batch 42 — app/filters/ (vwap_gate, early_session_disqualifier, mtf_bias, sd_zone_confluence, order_block_cache, liquidity_sweep)
Total Findings: 23 (0C / 5H / 11M / 7L)

🟡 Highs (5)
42.H-1 — vwap_gate.py: compute_vwap() recomputes VWAP from scratch on every call using all bars_session. Called once in _run_signal_pipeline() for the VWAP gate check, then called again for the liquidity sweep boost, then again from sniper.py's MTF bias block. Three full VWAP recomputations per signal on the same bar set. VWAP for a given bar set is deterministic — should be computed once and cached in thread_safe_state per ticker per session, or passed as a parameter.

42.H-2 — order_block_cache.py: cache_order_block() stores OBs in a module-level dict with no expiry. OBs identified at 9:35 persist through the entire session and across Railway restarts (they're in-memory only). apply_ob_retest_boost() checks if current price is within ob_low * 0.998 to ob_high * 1.002 — a ±0.2% tolerance. A $200 stock's OB has a ±$0.40 tolerance. By 14:00, price may be near an OB from 5 hours ago that is no longer structurally relevant. OBs should expire after MAX_WATCH_BARS bars (12 cycles = ~9 minutes) or when price moves more than 2× the OB range away.

42.H-3 — liquidity_sweep.py: apply_sweep_boost() checks if the previous bar swept below or_low (bull) or above or_high (bear) and then reclaimed. But or_high_ref and or_low_ref passed from sniper.py for the INTRADAY_BOS path are set to bos_signal["bos_price"] and zone_low/high — not the actual OR high/low. For INTRADAY_BOS signals, the sweep check is comparing against synthetic reference levels, not the true opening range. The boost fires based on incorrect reference levels for ~60% of signals (those using the intraday path).

42.H-4 — sd_zone_confluence.py: cache_sd_zones() is called on every process_ticker() call (before the regime filter and armed check). With 50 tickers per cycle, this recalculates and overwrites S/D zones for all 50 tickers every 45 seconds even for tickers that are already armed or will be filtered. Should only cache zones when the ticker passes the armed check and regime filter.

42.H-5 — mtf_bias.py: mtf_bias_engine.evaluate() is called inside _run_signal_pipeline() which fetches _bars_1m_raw = data_manager.get_bars_from_memory(ticker, limit=390) — a full day of 1m bars — just to resample to 15m and 1H. This is a duplicate DB read on top of the bars_session already in scope (39.H-3). A full 390-bar fetch on every signal for every ticker is one of the heaviest DB operations in the entire pipeline. Should resample from bars_session if it is 1m data, or accept pre-fetched bars.

42.M-6	early_session_disqualifier.py: should_skip_cfw6_or_early() takes or_range_pct and now_et but the function name and log message say "blocked before 9:45 AM" while the actual logic checks OR width, not time. Time gating and width gating are conflated.
42.M-7	vwap_gate.py: passes_vwap_gate() for bull requires entry_price > vwap. If entry_price == vwap (to the float precision), the gate fails. Should be >= vwap with a small tolerance (e.g., 0.05% of price).
42.M-8	order_block_cache.py: identify_order_block() returns the last opposing candle before bos_idx. With bos_idx being the last bar in bars_session, this is bars_session[-2] — the penultimate bar. For a fast gap-up open, this may be the pre-market bar, which has different liquidity characteristics than an intraday OB. Should filter out pre-9:30 bars from OB detection.
42.M-9	liquidity_sweep.py: The reclaim check close_reclaim for bull is prev_bar["close"] > or_low — any close above OR low counts as a reclaim. A close of $0.01 above OR low after sweeping $2 below is counted as a valid sweep+reclaim. Should require close to be above or_low + (or_high - or_low) * 0.20 (at least 20% back into the OR range). This is the same issue as Critical 5.G-18 from Batch 5 — confirming it is still unresolved.
42.M-10	sd_zone_confluence.py: Zone detection uses fixed lookback of last 50 bars. For a ticker with 300 bars by 14:00, this only looks back 50 minutes for S/D zones, missing significant daily structure. Should use configurable lookback with a default of 100 bars (covering morning session structure).
42.M-11	mtf_bias.py: record_stat() is called from sniper.py after evaluate() — but record_stat() modifies a module-level dict without a lock. Same threading concern as 40.M-12.
42.M-12	vwap_gate.py: VWAP_GATE_ENABLED is always True (hardcoded). No config toggle. If VWAP gate needs emergency disable (e.g., VWAP calculation bug), requires a code deploy. Should be VWAP_GATE_ENABLED = config.get("VWAP_GATE_ENABLED", True).
42.M-13	order_block_cache.py: apply_ob_retest_boost() returns (confidence, None) when no OB is cached for the ticker and (confidence + 0.03, ob_result) when a retest is detected. The 0.03 boost is hardcoded — should be config.OB_RETEST_BOOST for tunability. Same applies to S/D zone (+0.03) and sweep boosts.
42.M-14	sd_zone_confluence.py: clear_sd_cache() is called from EOD reset in sniper.py — correct. But cache_sd_zones() is called at the top of process_ticker() before the armed check. On a redeploy mid-session, cache_sd_zones() fires for all tickers before any signal history is loaded. Zones are computed on potentially incomplete bar data during the first post-redeploy cycle.
42.M-15	liquidity_sweep.py: apply_sweep_boost() returns (confidence, None) when no sweep is detected. sniper.py logs "No liquidity sweep detected" for every ticker every cycle when no sweep exists (the common case). This adds 50 log lines per cycle of noise. Should only log when a sweep IS detected.
42.M-16	early_session_disqualifier.py: The 9:45 AM early session gate is a return (hard skip) even for tickers with OR range above the threshold. should_skip_cfw6_or_early() blocks ALL OR signals before 9:45 regardless of OR quality. An A+ setup with a 3% OR that triggers a clean BOS at 9:32 is blocked until 9:45. Should only gate narrow-OR setups before 9:45, not wide ones.

42.L-17	All filter modules emit print("[MODULE] ✅ ...") at import. Collectively adds ~15 lines of startup noise. All should be logger.debug().
42.L-18	order_block_cache.py: clear_ob_cache(ticker=None) clears all OBs when ticker=None and one ticker's OB when specified. EOD reset in sniper.py calls clear_ob_cache() (no arg) — correct. But identify_order_block() is called mid-pipeline before cache_order_block(), so there's always a window where an OB is identified but not yet cached. Non-issue in practice (serial execution) but fragile.
42.L-19	vwap_gate.py: The VWAP formula sum(close * volume) / sum(volume) uses close not typical_price = (high + low + close) / 3. Standard VWAP uses typical price. Using close-only VWAP biases the result slightly upward in bull sessions and downward in bear sessions.
42.L-20	sd_zone_confluence.py: Zone labels ("supply" / "demand") are not standardized with direction ("bull" / "bear"). Downstream code must know to map demand → bull and supply → bear. Should add a direction field to zone dicts.
42.L-21	mtf_bias.py: evaluate() returns confidence_adj as a float boost value. sniper.py stores it as _mtf_bias_adj and adds it to final_confidence. The boost is unbounded from mtf_bias.py's perspective — should document the max/min range.
42.L-22	liquidity_sweep.py: apply_sweep_boost() accepts vwap=0.0 as default. A vwap=0.0 passed accidentally (e.g., compute_vwap() returns 0 on error) would make every price above VWAP, allowing spurious sweep reclaim detections. Should treat vwap <= 0 as "VWAP unavailable" and skip the VWAP-dependent sweep logic.
42.L-23	early_session_disqualifier.py: The threshold used in should_skip_cfw6_or_early() is compared against or_range_pct but the error message (39.L-24) says or_threshold — the variable names in the calling code (sniper.py) and this module are mismatched in semantics.
