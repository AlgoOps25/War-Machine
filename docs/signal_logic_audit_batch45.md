Total Findings: 16 (0C / 3H / 8M / 5L)

🟡 Highs (3)
45.H-1 — discord_helpers.py: send_simple_message() makes a synchronous HTTP POST to the Discord webhook URL inside process_ticker(). This blocks the scan loop for the duration of the HTTP request (typically 100–500ms, up to 30s on timeout). During the OR open with multiple tickers firing simultaneously, concurrent Discord calls from the serial executor block each subsequent ticker scan. Discord webhook latency directly degrades scan cycle time. Should be fire-and-forget via threading.Thread or added to a queue processed by a dedicated Discord sender thread.

45.H-2 — discord_helpers.py: The webhook URL is read from os.getenv("DISCORD_WEBHOOK_URL") on every call (not cached at module load). On Railway, env vars are stable — but every call does an os.getenv() lookup. More critically: if the env var is unset, os.getenv() returns None and the subsequent requests.post(None, ...) raises TypeError. validate_required_env_vars() in scanner.py checks for this at startup — but discord_helpers.py is also imported by sniper.py before validate_required_env_vars() runs. Should cache: _WEBHOOK_URL = os.getenv("DISCORD_WEBHOOK_URL") at module load and raise ValueError immediately if None.

45.H-3 — screening/ modules: get_ticker_screener_metadata() (the stub fallback) returns {'qualified': False, 'score': 0, 'rvol': 0.0, 'tier': None} — always qualified: False. With screener_integration unavailable (the module was deleted per sniper.py Phase comments), the explosive mover override in process_ticker() never fires — every ticker returns qualified: False. The regime filter can never be bypassed. This means on high-VIX days where the regime filter blocks scanning, no explosive movers are ever detected and no trades are taken. The stub should return qualified: True for tickers with RVOL above a threshold derived from the live bar data already in scope.

45.M-4	discord_helpers.py: No rate limiting. arm_signal.py sends a Discord message on every signal arm. scanner.py sends a Discord message on startup, reconnect, EOD. sniper.py sends BOS watch alerts. No global rate limiter exists — if 10 tickers fire simultaneously, 10 webhook posts hit Discord in ~1 second. Discord's webhook rate limit is 30 requests per 30 seconds per webhook. Exceeding it returns HTTP 429 which should be caught and retried.
45.M-5	discord_helpers.py: send_simple_message() timeout is likely the requests default (no explicit timeout set) — meaning it can block indefinitely on a slow Discord response. Should set timeout=5 seconds.
45.M-6	screening/screener_integration.py (deleted): The stub get_ticker_screener_metadata() in sniper.py never calls any live data source. RVOL, score, and tier are all zero/None. These values are used for the explosive mover override AND for the metadata dict passed to arm_ticker() and recorded in the DB. Trade records will have rvol=0.0 for all entries, permanently corrupting the performance analytics that depend on RVOL for signal quality attribution.
45.M-7	discord_helpers.py: Message formatting uses plain text. Long ticker watchlists in the pre-market alert exceed Discord's 2000-character message limit. No truncation or pagination logic. Messages over 2000 chars are silently rejected by Discord (HTTP 400). Should truncate with "... (truncated)" at 1900 chars.
45.M-8	screening/ directory: Without screener_integration.py, the entire upstream screening funnel (EODHD screener → tier assignment → RVOL calculation) is dark. The get_screener_tickers() function in scanner.py (38.M-6 — dead code) was the last remnant of this path. The screening layer is effectively absent from production.
45.M-9	discord_helpers.py: send_simple_message() does not include a Content-Type: application/json header explicitly — relies on requests to set it via json= parameter (which it does). Low risk but should be explicit.
45.M-10	discord_helpers.py: Error handling catches Exception and prints the error. On Railway, a Discord webhook failure during a signal arm means the trade fired (position opened) but the Discord alert was silently dropped. The trader has no notification that a position was opened. Should escalate to a secondary notification path (email, SMS) on Discord failure, or at minimum log to Railway with logger.critical().
45.M-11	screening/: No screener means no pre-market watchlist filtering based on RVOL, gap%, or float. The watchlist is likely sourced from a static config list. Without dynamic screening, War Machine scans the same tickers regardless of which ones are moving that day.
