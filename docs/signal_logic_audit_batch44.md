Batch 44 — utils/config.py + utils/time_helpers.py + utils/production_helpers.py
Total Findings: 17 (0C / 3H / 9M / 5L)

🟡 Highs (3)
44.H-1 — config.py: MIN_CONFIDENCE_BY_GRADE and CONFIDENCE_ABSOLUTE_FLOOR are read from environment variables with float(os.getenv(..., default)). If any env var is set to a non-numeric string (e.g., MIN_CONFIDENCE_OR="" accidentally), float("") raises ValueError at import time — killing the entire process before any error handling is in place. validate_required_env_vars() (Phase 1.28 in scanner.py) only checks for the presence of 3 keys (EODHD_API_KEY, DATABASE_URL, DISCORD_WEBHOOK_URL), not for validity of numeric config values. Should wrap all float(os.getenv(...)) calls in try/except ValueError with a fallback to the default.

44.H-2 — time_helpers.py: _now_et() returns datetime.now(ET) — correct. But _strip_tz() does dt.replace(tzinfo=None) — drops timezone info without converting to ET first. If dt is UTC-aware (e.g., from a Postgres TIMESTAMPTZ column which returns UTC in psycopg2), _strip_tz(dt) returns a naive datetime with the UTC value, not the ET value. Every comparison of _strip_tz(db_timestamp) with _now_et().replace(tzinfo=None) compares UTC-naive against ET-naive — a 4-5 hour discrepancy. This is the root cause of the tz-naive bug family that spans 15+ findings across batches (37.H-1, 38.M-1, 39.H-1, etc.). Fix _strip_tz() to convert to ET before stripping: dt.astimezone(ET).replace(tzinfo=None).

44.H-3 — production_helpers.py: _fetch_data_safe() has a retry loop with time.sleep(1) between retries. Called from process_ticker() inside the ticker scan loop. A 3-retry fetch with 1-second sleeps = up to 3 seconds of blocking sleep per ticker failure. With 50 tickers, if all fetches fail (e.g., DB pool exhaustion), the scan cycle blocks for up to 150 seconds — 2.5 minutes of dead time at the OR open. Should use exponential backoff with a maximum of 0.5 seconds total (not per retry) during the scan cycle.

If final_grade = "B" and "B" is not in MIN_CONFIDENCE_BY_GRADE, config.MIN_CONFIDENCE_BY_GRADE.get(final_grade, min_type) falls back to min_type (correct). 44.M-7	time_helpers.py: _bar_time(bar) returns bar["datetime"].time() if datetime has a .time() method, else returns datetime directly. If bar["datetime"] is already a time object (not datetime), hasattr(bt, "time") is True because time objects have a time() method — wait, datetime.time objects do NOT have a .time() method. Only datetime.datetime objects have .time(). So the check is correct. However _bar_time() returns None if bar.get("datetime") is None — callers must handle None return. Several call sites do not check for None return.
44.M-8	production_helpers.py: _db_operation_safe() wraps DB operations with retry and fallback. The fallback returns None — but some DB operations (e.g., get_session_status()) return a dict. Callers that don't check for None return and immediately access result["daily_stats"] raise TypeError. Should document the return type contract and ensure all callers handle None.
44.M-9	config.py: ORB_BREAK_THRESHOLD = float(os.getenv("ORB_BREAK_THRESHOLD", "0.001")) (0.1%). This threshold applies to both the main OR path and the MTF compression path in mtf_integration.py. As noted in 40.M-10, this threshold is too permissive for compressed lower-TF bars. Should have separate MTF_ORB_BREAK_THRESHOLD for MTF scanning.
44.M-10	time_helpers.py: _strip_tz() is called in 20+ places across the codebase with UTC datetimes from the DB. Until 44.H-2 is fixed, every one of these call sites has a latent 4–5 hour offset bug. The blast radius of this fix is wide — requires testing every time comparison in the pipeline.
44.M-11	production_helpers.py: _send_alert_safe() catches Exception and returns False. If Discord webhook is down, alerts are silently dropped with no local logging of what alert was lost. Should log the alert body to logger.error() when webhook fails so the alert content is preserved in Railway logs.
44.M-12	config.py: Config values are loaded at module import. On Railway, env vars are set before the process starts — correct. But any os.environ changes after import (e.g., in tests using monkeypatch.setenv) don't affect already-loaded config values. Tests that need to override config must reload the module. Should document this constraint or use os.getenv() at call time for testability.
