Batch 43 — app/validation/greeks_precheck.py + app/signals/vwap_reclaim.py + app/signals/signal_generator_cooldown.py
Total Findings: 18 (0C / 4H / 9M / 5L)

🟡 Highs (4)
43.H-1 — greeks_precheck.py: validate_signal_greeks() fetches a live options chain on every call. Called from inside _run_signal_pipeline() with OPTIONS_PRE_GATE_MODE = "HARD". With 50 tickers per cycle, this is 50 live options chain fetches (Tradier API calls) per scan cycle. Tradier's sandbox/paper rate limit is ~120 requests/minute; production is higher but not unlimited. At the OR open with 50 tickers all fetching chains simultaneously, rate limiting will cause timeouts that are caught by the except Exception handler and result in greeks_valid = False — silently dropping every signal during the busiest period. Should cache the chain per ticker for the current scan cycle (60-second TTL).

43.H-2 — greeks_precheck.py: validate_signal_greeks() uses _proxy_entry = bars_session[-1]["close"] as the entry price for strike selection. At the time of the greeks pre-check, the signal hasn't been confirmed yet — the actual entry price will be the next bar's open (post-confirmation). Using the current close for strike selection means the selected strike may not be the ATM strike at entry. For a $200 stock moving $1-2 per bar, this results in the wrong strike being checked and potentially the wrong IV/delta/theta being validated.

43.H-3 — signal_generator_cooldown.py: is_on_cooldown() queries the signal_cooldowns DB table on every call. With OPTIONS_PRE_GATE_MODE = "HARD", is_on_cooldown() is the very first check in _run_signal_pipeline() — called for every ticker that reaches the pipeline. This is a synchronous DB query in the hot path. Should maintain an in-memory cache of active cooldowns that is populated at startup from the DB and updated on set_cooldown(), falling back to DB only on cache miss.

43.H-4 — vwap_reclaim.py: detect_vwap_reclaim() defines its own VWAP computation inline rather than using vwap_gate.compute_vwap(). Two different VWAP implementations now exist in the codebase. If vwap_gate.compute_vwap() is fixed (e.g., to use typical price per 42.L-19), vwap_reclaim.py's inline version remains broken. Should import and use compute_vwap from vwap_gate.py.

🟠 Mediums (9)
ID	Issue
43.M-5	greeks_precheck.py: Greeks thresholds (min delta, max theta, max IV) are hardcoded constants. No config reference. A day with unusually high IV (earnings season) will fail the max IV check and block all signals. Should be config.GREEKS_MAX_IV etc.
43.M-6	signal_generator_cooldown.py: set_cooldown() inserts into signal_cooldowns with direction as a column. is_on_cooldown() queries WHERE ticker = %s AND direction = %s. A bull cooldown does not block a bear signal — correct for reversal trades. But the reversal_window logic (15-min block after a trade) should block the opposite direction too. The current implementation only blocks same-direction re-entries within same_direction_window (30 min) and doesn't enforce the reversal window cross-direction block (see also 39.H-6).
43.M-7	vwap_reclaim.py: detect_vwap_reclaim() checks if the previous bar crossed VWAP and the current bar confirmed. Returns reclaim_bar_idx as the index in bars_session. But sniper.py passes this index as breakout_idx to _run_signal_pipeline() which uses it for find_fvg_after_bos(). A VWAP reclaim bar index from bars_session (which may be 5m data) passed to find_fvg_after_bos() which scans from bos_idx - 5 forward — the FVG search starts 5 bars before the VWAP reclaim, potentially finding a pre-reclaim FVG.
43.M-8	greeks_precheck.py: When OPTIONS_PRE_GATE_MODE = "SOFT", a failed greeks check is logged but the pipeline continues. The full options_filter.validate_signal_for_options() is then called again in the soft path. Two options chain fetches occur for the same ticker in soft mode — one for greeks pre-check, one for full validation.
43.M-9	signal_generator_cooldown.py: is_on_cooldown() cleans up expired cooldowns on every query: DELETE FROM signal_cooldowns WHERE expires_at < NOW(). Running a DELETE on every cooldown check adds unnecessary write load during peak signal generation. Should clean up in a scheduled task (e.g., hourly) rather than on every read.
43.M-10	vwap_reclaim.py: The synthetic FVG zone vwap * 0.9985 to vwap * 1.0015 (±0.15%) is hardcoded (same as 39.M-17). For high-volatility tickers, this zone is too narrow and will rarely be retested. For low-volatility tickers, it's still too narrow. Should use get_adaptive_fvg_threshold().
43.M-11	greeks_precheck.py: The _pre_options_data from the greeks pre-check is passed back through sniper.py as options_rec for multiplier extraction. If greeks pre-check passes but the full options_filter.validate_signal_for_options() also runs and returns different gex_multiplier values, sniper.py uses _pre_options_data (the pre-check result) for multipliers, not the full validation result. The more complete data is discarded.
43.M-12	signal_generator_cooldown.py: set_cooldown() uses datetime.utcnow() + timedelta(minutes=window) for expires_at. On Railway (UTC), this is correct. But if the DB column is TIMESTAMPTZ, comparing with NOW() in the cleanup DELETE is fine. If it's TIMESTAMP (no TZ), NOW() in Postgres returns the session TZ (which may be UTC or local), introducing drift. Should explicitly use NOW() AT TIME ZONE 'UTC' in the SQL.
43.M-13	vwap_reclaim.py: detect_vwap_reclaim() returns the reclaim signal without checking if the ticker is already in watching_signals or armed_signals. A ticker currently in watch mode for a BOS signal can simultaneously fire a VWAP reclaim signal — two pipeline calls for the same ticker in the same cycle (one from the watch path, one from the VWAP reclaim path at the bottom of process_ticker()). The armed check at the top of process_ticker() prevents double-arming, but two pipeline calls still run all gates twice.

43.L-14	greeks_precheck.py: Import-time print() statements. Should be logger.debug().
43.L-15	vwap_reclaim.py: detect_vwap_reclaim() returns None when the bar count is too low. The sniper.py caller prints "No VWAP reclaim signal" on every None return — adds a log line every cycle per ticker. Should only log when a reclaim IS detected.
43.L-16	signal_generator_cooldown.py: The table name signal_cooldowns is hardcoded in SQL strings throughout. Should be a module-level constant COOLDOWN_TABLE = "signal_cooldowns".
43.L-17	greeks_precheck.py: validate_signal_greeks() logs the reason string from the API response which may contain ticker symbol and price data. On Railway, these logs are visible in the Railway dashboard — no PII risk, but verbose.
43.L-18	vwap_reclaim.py: The vr_or_high = entry_price * 1.005 and vr_or_low = entry_price * 0.995 synthetic OR levels (±0.5%) are wider than the FVG zone (±0.15%) — the "OR" reference levels are larger than the signal zone itself, which inverts the usual relationship. This may cause compute_stop_and_targets() to compute incorrect stops.
