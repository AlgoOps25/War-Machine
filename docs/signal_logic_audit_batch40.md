🟡 Highs (4)
40.H-1 — find_fvg_after_bos() searches backwards from bos_idx - 5. The comment says "look back a few bars too" but scanning bars before the BOS for the FVG violates the BOS+FVG logic. A FVG that formed before the BOS is a pre-break imbalance, not a post-BOS retracement zone. On a fast-moving ticker the FVG at bos_idx - 3 may already be miles above the current retracement. The search_start = max(0, bos_idx - 5) should be search_start = bos_idx — scan forward only.

40.H-2 — detect_bos() only fires on the very last bar. detect_bos(bars) computes structure on bars[:-1] and checks only bars[-1] (the current bar). If the BOS occurred 2 bars ago and the FVG has since formed but bars[-1] is inside a pullback (close between swing_high and swing_low), detect_bos() returns None and the entire signal is missed. The intraday BOS path in sniper.py calls scan_bos_fvg() on every cycle — if the BOS bar was cycle N and the FVG forms in cycle N+2, cycle N+1 returns None (no BOS on latest bar) and cycle N+2 also returns None. The BOS is permanently undetected. The watch-signal path in sniper.py exists to solve this but only for the OR path, not INTRADAY_BOS. detect_bos() should scan the last N bars (e.g., last 5) and return the most recent BOS, not just the single latest.

40.H-3 — check_fvg_entry() bull bounce check: prev_bar["close"] >= fvg_mid. If the FVG zone is $10.00–$10.10 (mid=$10.05) and the confirmation bar closes exactly at $10.05, it passes. But fvg_mid is computed as (fvg_low + fvg_high) / 2 from find_fvg_after_bos() where fvg_high = c2["low"] and fvg_low = c0["high"]. For a bull FVG, price must bounce above the FVG (close above the top of the gap = fvg_high), not just above the midpoint. Closing at the midpoint means price is still inside the gap — not a confirmed bounce. Should be prev_bar["close"] >= fvg["fvg_high"] for bull, prev_bar["close"] <= fvg["fvg_low"] for bear.

40.H-4 — mtf_integration.py: _mtf_cache is keyed by f"{ticker}_{direction}" and cleared only on date rollover. If enhance_signal_with_mtf(NVDA, "bull", bars_9:35) is called at 9:35 and cached, then at 11:00 when NVDA has 90 more bars, enhance_signal_with_mtf(NVDA, "bull", bars_11:00) returns the stale 9:35 result. The cache never expires intra-day. MTF convergence at 9:35 (5 OR bars) is completely different from convergence at 11:00 (90 bars). The cache should either be keyed by (ticker, direction, len(bars)) or expire after 2 scan cycles.

40.M-5	find_swing_points() uses bars[-(lookback * 3):] — the most recent lookback * 3 = 30 bars. With LOOKBACK_SWING=10, this is the last 30 bars. Early session (9:30–9:45 = 15 bars), the full session is used but the scan window range(lookback//2, len(recent) - lookback//2) becomes range(5, 25) — correct. However swing_high is updated to the maximum high seen across all candidate positions, not the most recent swing high. In a trending session with multiple swing highs at 9:40 ($100), 10:00 ($102), 10:30 ($105), the returned swing_high is $105 (the largest) but the structurally relevant swing high for BOS detection is $105 too (same) — however if the trend is down after $105, the relevant high is the last swing high, which may be $103. Should return most recent, not highest.
40.M-6	classify_confirmation_candle() bull lower_wick calculation: lower_wick = o - l if c >= o else c - l. For a green candle (c >= o), lower wick = open - low. For a red candle (c < o), lower wick = close - low. This is correct. But the A+ grade check is_green and (lower_wick/total_range) < 0.20 uses o - l as lower wick for green candles — this includes the entire space from open down to low, which is not the lower wick (it's open minus low). Lower wick for a green candle should be open - low only if open > low, which is min(open, close) - low. Should use min(o, c) - l.
40.M-7	scan_tf_for_signal() in mtf_integration.py uses config.FVG_MIN_SIZE_PCT directly (not the adaptive threshold from get_adaptive_fvg_threshold()). The adaptive threshold passed from sniper.py to scan_bos_fvg() is ignored entirely in the MTF path — all lower-TF scans use the static config value. Should pass the adaptive threshold down.
40.M-8	scan_tf_for_signal() searches for FVG confirmation in range(breakout_idx + 3, min(breakout_idx + 10, len(bars))) — a fixed 7-bar window post-breakout. For 1m bars (1m compressed from 5m), 7 bars = 7 minutes. For 3m bars, 7 bars = 21 minutes. The same fixed window is used regardless of timeframe, making it more restrictive on higher timeframes and more permissive on lower. Should be proportional or time-based.
40.M-9	compute_or() in mtf_integration.py uses time(9, 30) <= bar_time < time(9, 40) — a 10-minute OR window on the compressed bars. On 3m bars, bars at 9:30, 9:33, 9:36, 9:39 are included (4 bars). On 2m bars, 5 bars. On 1m bars, 10 bars. But bos_fvg_engine.py's compute_opening_range_from_bars() (called from sniper.py) uses a 9:45 window (Phase 2 widening). The MTF OR window is 5 minutes shorter than the main OR — creating mismatched OR levels between the two systems.
40.M-10	detect_breakout() in mtf_integration.py uses config.ORB_BREAK_THRESHOLD for all timeframes. This is likely a percentage threshold (e.g., 0.001 = 0.1%). Applied to compressed 3m/2m/1m bars, the same threshold is used regardless of whether the compressed bar already represents an aggregated move. A 0.1% threshold on a 5m bar that aggregates five 1m bars may already be satisfied by noise.
40.M-11	run_mtf_trend_step() applies confidence = min(0.99, confidence + boost) — a hard cap of 0.99, not config.CONFIDENCE_ABSOLUTE_CEILING (0.95 everywhere else in the codebase). Confidence can reach 0.99 from this step alone if boost is large, bypassing the 0.95 ceiling enforced in _run_signal_pipeline(). The final clamp in _run_signal_pipeline() catches it, but the intermediate value at 0.99 affects the gate calculation if any intermediate check uses confidence before the final clamp. Should be min(0.95, confidence + boost).
40.M-12	_mtf_stats is a module-level mutable dict with no thread lock. enhance_signal_with_mtf() is called from process_ticker() which runs across multiple threads (if 38.H-2 is ever fixed to use max_workers > 1). Concurrent increments to _mtf_stats['analyzed'] etc. are not atomic. Should use a threading.Lock or collections.Counter with atomic ops.
40.M-13	compute_0dte_stops_and_targets() sets rr_t1: 1.5 and rr_t2: 2.5 as fixed constants. But sniper.py calls compute_stop_and_targets() from app.risk.trade_calculator (not this function) for the actual stop/target calculation. This function is only used internally by scan_bos_fvg() — but scan_bos_fvg()'s returned stop, t1, t2 values are never used by sniper.py (sniper.py overwrites them with compute_stop_and_targets()). The stop/t1/t2 fields in the scan_bos_fvg() return dict are dead data.
